# 跨层材料搬运系统设计文档

## 问题

RimWorld 原版 job 系统假设材料和目标在同一张地图上。多层框架下，材料可能在 A 层，目标（蓝图/加油口/工作台等）在 B 层，原版 WorkGiver 找不到材料就放弃了。

### 受影响的 Job 类型

| 类型 | WorkGiver | 目标 | 材料 |
|------|-----------|------|------|
| 建造 | ConstructDeliverResourcesToBlueprints/Frames | Blueprint / Frame | 建材（木头、钢铁等） |
| 加油 | Refuel | CompRefuelable 建筑 | 燃料（木头、化合燃料等） |
| 制作 | DoBill | 工作台（有激活的 Bill） | 配方原料 |
| 喂囚犯 | Warden_Feed | 囚犯 | 食物 |
| 喂病人 | FeedPatient | 病人 | 食物 |
| 安装 | Install | Blueprint_Install | 待安装物品 |
| 搬尸体 | HaulCorpses | 坟墓/火化台 | 尸体 |
| 合并堆叠 | Merge | 已有堆叠 | 同类物品 |

### 共同模式

所有这些 job 本质上都是：

```
找到 item → pawn 拿起来 → 送到 building
```

跨层时只需在中间插入走楼梯：

```
pawn → item → 拿起 → 楼梯 → 转移 → building → 交付
```

---

## 现有实现的问题

### 当前方案

1. `TryCrossLevelScan`：传送 pawn 到其他层，让原版 WorkGiver 扫描
2. `TryCrossLevelConstructionFetch`：扫描后兜底，找其他层的框架
3. `Patch_ConstructDeliverResources`：patch 原版建造材料查找

### 问题

- `TryCrossLevelScan` 期间 `Scanning=true`，阻断了 `Patch_ConstructDeliverResources`
- `TryCrossLevelConstructionFetch` 只搜索 `BuildingFrame`，漏掉了 `Blueprint`
- 每种 job 类型需要单独写 patch，代码重复且维护成本高
- 建造以外的 job（加油、制作等）完全没有跨层支持

---

## 新方案：三层架构

### 层级总览

```
┌─────────────────────────────────────────┐
│  第一层：全局需求注册表（透视层）          │
│  "谁需要什么材料"                         │
├─────────────────────────────────────────┤
│  第二层：材料匹配（坐标层）               │
│  "材料在哪，目标在哪"                     │
├─────────────────────────────────────────┤
│  第三层：任务分配与执行（执行层）          │
│  "pawn 怎么跨层搬运"                      │
└─────────────────────────────────────────┘
        ↕
  原版 WorkGiver 优先级体系（不改动）
```

### 第一层：全局需求注册表（GlobalMaterialRegistry）

维护一张"谁需要什么"的表，事件驱动更新。

```csharp
public static class GlobalMaterialRegistry
{
    // key = target Thing 的 thingIDNumber
    private static Dictionary<int, MaterialNeed> needs;

    public struct MaterialNeed
    {
        public Thing target;           // 蓝图/加油口/工作台等
        public Map targetMap;
        public int targetElevation;
        public NeedType type;          // Construction, Refuel, Bill, Feed...
        public List<ThingDefCount> materials;  // 需要的材料列表
    }

    // 注册需求（蓝图放下、refuelable 缺燃料、bill 激活时调用）
    public static void Register(Thing target, NeedType type, List<ThingDefCount> materials);

    // 注销需求（蓝图取消、建筑完成、燃料满了时调用）
    public static void Unregister(Thing target);

    // 更新需求（材料部分送达时调用）
    public static void UpdateCount(Thing target, ThingDef material, int newCount);

    // 查询：获取指定楼层以外的所有需求
    public static List<MaterialNeed> GetNeedsExcluding(Map excludeMap);

    // 查询：获取指定材料类型相关的需求
    public static List<MaterialNeed> GetNeedsForMaterial(ThingDef materialDef);
}
```

#### 注册时机（Harmony Patch 或事件 Hook）

| 事件 | 动作 |
|------|------|
| Blueprint/Frame 生成 (SpawnSetup) | Register(target, Construction, TotalMaterialCost) |
| Blueprint/Frame 销毁 (DeSpawn) | Unregister(target) |
| 材料送达 (IConstructible 材料变化) | UpdateCount |
| CompRefuelable.Fuel < FuelCapacity | Register(target, Refuel, fuelDef) |
| CompRefuelable 满了 | Unregister(target) |
| Bill 激活且有原料需求 | Register(target, Bill, ingredients) |
| Bill 完成/暂停 | Unregister(target) |

#### 性能考虑

- 事件驱动，不做定时全量扫描
- Dictionary 按 thingIDNumber 索引，O(1) 查找
- 需求变化频率低（蓝图不会每 tick 变），注册/注销开销可忽略
- 可选：按 materialDef 建二级索引，加速"本层有 X 材料，谁需要？"的查询

### 第二层：材料匹配（MaterialMatcher）

在 pawn 找工作时，将注册表中的需求与各层的材料库存匹配。

```csharp
public static class MaterialMatcher
{
    public struct MatchResult
    {
        public MaterialNeed need;       // 来自注册表
        public Thing material;          // 找到的材料
        public Map materialMap;         // 材料所在地图
        public int materialElevation;
        public int distance;            // 层级距离（用于排序）
    }

    /// <summary>
    /// 为 pawn 查找可执行的跨层材料搬运任务。
    /// 只返回 pawn 当前层有材料、目标在其他层的匹配。
    /// </summary>
    public static List<MatchResult> FindMatches(Pawn pawn)
    {
        // 1. 从注册表获取其他层的需求
        // 2. 对每个需求，检查 pawn 当前层是否有所需材料
        // 3. 检查材料是否可达、可预约
        // 4. 检查是否有楼梯通往目标层
        // 5. 按层级距离排序返回
    }
}
```

#### 匹配策略

```
场景 A：pawn 在材料层，目标在其他层
  → FindMatches 返回匹配
  → pawn 拿材料 → 走楼梯 → 送到目标

场景 B：pawn 在目标层，材料在其他层
  → 由现有 TryCrossLevelScan 处理（传送扫描）
  → 或：FindMatches 反向查询（本层有目标，其他层有材料）
  → pawn 走楼梯 → 拿材料 → 走楼梯回来 → 送到目标

场景 C：pawn、材料、目标各在不同层
  → 暂不处理（复杂度高，收益低）
  → 或拆成两步：先把材料搬到目标层，再本地交付
```

### 第三层：任务分配与执行

#### 接入点

在 `JobGiver_Work.TryIssueJobPackage` Postfix 中（现有 `Patch_CrossLevelJobScan` 位置）：

```
原版 WorkGiver 扫本层 → 有活 → 正常执行
                      → 没活 ↓
TryCrossLevelScan（现有逻辑，处理目标和材料在同一层的情况）
                      → 找到 → 走楼梯去干活
                      → 没找到 ↓
MaterialMatcher.FindMatches(pawn)
                      → 有匹配 → 生成跨层搬运 job
                      → 无匹配 → 真的没活
```

#### 通用跨层搬运 JobDriver

复用/扩展现有的 `JobDriver_ReturnWithMaterial`：

```csharp
public class JobDriver_CrossLevelDeliver : JobDriver
{
    // 通用流程：
    // 1. 从 FetchData 读取 materialDef + target + targetElevation
    // 2. 在本层找到材料（GenClosest）
    // 3. 走到材料 → 拿起
    // 4. 走到楼梯 → 转移到目标层
    // 5. 根据 NeedType 执行交付：
    //    - Construction → HaulToContainer (送到蓝图/框架)
    //    - Refuel → Refuel job
    //    - Bill → DoBill 的材料交付
    //    - Feed → 喂食
}
```

#### FetchData 扩展

```csharp
public struct FetchData
{
    public ThingDef thingDef;       // 材料类型
    public Thing target;            // 目标（蓝图/加油口/工作台等）
    public int returnElevation;     // 目标所在层
    public NeedType needType;       // 需求类型（决定交付方式）
    public int count;               // 需要搬运的数量
}
```

## 与现有系统的关系

### 保留

- `TryCrossLevelScan`：处理"目标和材料在同一层，pawn 在别的层"的情况
- `MLF_UseStairs` / `JobDriver_UseStairs`：楼梯转移逻辑
- `StairTransferUtility`：pawn 转移工具
- `CrossLevelJobUtility`：冷却、延迟 job、楼梯查找等工具方法
- 跨层需求扫描（吃饭、睡觉、娱乐等）：这些不涉及材料搬运，保持现有逻辑

### 替换

- `TryCrossLevelConstructionFetch`：被 MaterialMatcher 替代
- `Patch_ConstructDeliverResources`：被 GlobalMaterialRegistry + 通用 JobDriver 替代
- 未来不需要为每种 job 单独写 patch

### 新增

- `GlobalMaterialRegistry`：全局需求注册表
- `MaterialMatcher`：材料匹配器
- `JobDriver_CrossLevelDeliver`：通用跨层搬运 JobDriver
- 注册/注销的 Harmony Patch（Blueprint.SpawnSetup、CompRefuelable 等）

## 实施步骤建议

### 阶段 1：修复当前建造 bug（快速）
- `TryCrossLevelConstructionFetch` 加上 Blueprint 搜索
- 验证建造跨层可用

### 阶段 2：搭建注册表框架
- 实现 `GlobalMaterialRegistry`
- 先只注册 Construction 类型（Blueprint/Frame）
- 用注册表替换 `TryCrossLevelConstructionFetch` 的扫描逻辑

### 阶段 3：通用化 JobDriver
- 扩展 `FetchData` 支持 NeedType
- 实现 `JobDriver_CrossLevelDeliver` 的通用交付逻辑

### 阶段 4：扩展到其他 job 类型
- 逐步添加 Refuel、DoBill、Feed 等的注册 hook
- 每添加一种，测试验证

## 开放问题

1. **Bill 的材料需求动态性高**：工作台的 bill 可能随时暂停/恢复，原料需求取决于当前库存和配方，注册时机需要仔细考虑
2. **多 pawn 竞争**：两个 pawn 同时看到同一个跨层需求，需要预约机制防止重复搬运
3. **场景 B 的处理**：pawn 在目标层但材料在其他层，是走两趟楼梯还是等材料被搬过来？
4. **优先级排序**：跨层任务应该比本层任务优先级低吗？还是按原版 WorkGiver 优先级？
5. **与 Digital Storage 等 mod 的兼容**：如果材料已经被数字化存储，注册表需要感知吗？
